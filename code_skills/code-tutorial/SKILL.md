# Code Tutorial Skill

深度阅读当前项目代码，自动生成高质量的项目新手入门教程。

---

## 工作流程

### Step 1：项目全局概览

首先快速扫描项目根目录，建立整体认知：

1. 读取项目根目录文件列表（`ls`），识别项目类型和技术栈
2. 读取以下关键文件（存在即读）：
   - `README.md` / `README` — 项目说明
   - `package.json` / `go.mod` / `pom.xml` / `Cargo.toml` / `pyproject.toml` / `requirements.txt` / `Gemfile` — 依赖与元信息
   - `Makefile` / `Justfile` / `Taskfile.yml` — 构建与任务
   - `Dockerfile` / `docker-compose.yml` — 部署方式
   - `.env.example` / `config.*` — 配置项
   - `CLAUDE.md` / `CONTRIBUTING.md` — 开发规范
3. 读取 CI/CD 配置（`.github/workflows/`、`.gitlab-ci.yml` 等），了解自动化流程

> 目标：用 1-2 分钟确定项目语言、框架、构建工具、入口文件。

---

### Step 2：代码结构分析

递归扫描目录结构（排除 `node_modules`、`vendor`、`.git`、`dist`、`build` 等），然后：

1. **识别目录分层**：`src/`、`cmd/`、`internal/`、`pkg/`、`lib/`、`app/`、`api/`、`routes/`、`models/`、`services/`、`components/` 等
2. **定位入口文件**：`main.*`、`index.*`、`app.*`、`server.*`、`cmd/*/main.go` 等
3. **识别核心模块**：按目录名和文件数量判断哪些是核心模块
4. **读取入口文件**，追踪启动流程和主要依赖关系

> 目标：画出项目的"骨架"——哪些目录是核心的，它们之间什么关系。

---

### Step 3：深入阅读核心代码

按优先级逐层深入：

| 优先级 | 阅读目标 | 说明 |
|--------|----------|------|
| 🔴 必读 | 入口文件 | 启动流程、初始化逻辑、依赖注入 |
| 🔴 必读 | 核心业务模块 | 项目最重要的 3-5 个模块/包 |
| 🔴 必读 | 数据模型 | models / types / schema / proto 定义 |
| 🟡 重要 | API 层 | routes / handlers / controllers / resolvers |
| 🟡 重要 | 配置与中间件 | 配置加载、鉴权、日志、错误处理 |
| 🟢 补充 | 工具函数 | utils / helpers / common |
| 🟢 补充 | 测试代码 | 测试用例能揭示预期行为和边界条件 |

每个模块重点关注：
- 对外暴露的接口/函数签名
- 核心数据结构
- 关键业务逻辑流程
- 模块间的依赖关系

> 目标：阅读 **15-30 个核心文件**，覆盖项目的主要功能。

---

### Step 4：梳理知识，构建教程大纲

阅读完成后，整理以下信息：

1. **这个项目是做什么的？** 一句话描述核心功能
2. **技术栈是什么？** 语言、框架、数据库、中间件等
3. **代码是怎么组织的？** 目录结构和分层逻辑
4. **启动流程是怎样的？** 从 main 函数到服务就绪的完整链路
5. **核心模块有哪些？** 每个模块的职责和相互关系
6. **关键数据流是怎样的？** 一个典型请求从入口到返回的完整路径
7. **如何本地跑起来？** 环境依赖、配置、启动命令
8. **如何开发调试？** 开发模式、热重载、日志查看
9. **代码规范和约定？** 命名风格、分层规则、提交规范

## 质量要求

输出教程时，必须满足：

- ✅ **基于真实代码**：所有分析必须来自实际代码阅读，不能凭猜测，引用代码时标注 `文件路径:行号`
- ✅ **结构要清晰**：遵循"整体 → 局部 → 实操"的认知顺序，先建立全局观再深入细节
- ✅ **用人话解释**：避免照搬代码注释，用自己的理解重新表述，必要时用类比
- ✅ **架构图要直观**：用 ASCII 图或分层描述展示模块关系和数据流
- ✅ **启动步骤可执行**：本地开发指南必须完整到可以直接复制粘贴运行
- ✅ **代码位置要精确**：引用代码时使用 `文件路径:行号` 格式，方便跳转
- ✅ **数据流要完整**：至少追踪一个典型场景的完整请求链路
- ✅ **新手友好**：提供具体的上手任务建议，而不是空泛的"看看代码"
- ✅ **中文友好**：默认输出中文，技术术语/命令/代码保持英文原文

## 特殊情况处理

**项目规模很大（>1000 文件）**：
- 先问用户关注哪个模块/功能，聚焦该部分生成教程
- 在教程开头说明"本教程聚焦 XXX 模块"

**Monorepo（多项目）**：
- 先识别各子项目，询问用户关注哪个
- 或生成一份总览教程，概述各子项目职责和关系

**代码注释很少**：
- 更多依赖代码结构、命名、测试用例来推断意图
- 在教程中标注"以下为代码分析推断，建议与团队确认"

**项目缺少 README 或文档**：
- 完全依赖代码分析，教程中标注"本项目暂无文档，以下基于代码分析"
- 在最后建议补充文档

**用户指定了关注范围**：
- 如果用户说"帮我看看认证模块"或"我只关心 API 层"，则聚焦该部分
- 但仍需先做全局概览以提供上下文
